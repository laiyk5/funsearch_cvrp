\begin{abstract}
We present FunSearch for CVRP heuristic discovery, evolving priority and savings functions with LLMs from 7B to frontier scale. A controlled comparison shows the algorithmic skeleton dominates LLM capability: a savings spec achieves 1.68\% gap while a priority spec plateaus at 11.65\% with the same 7B model. Train/test/val splits with a dynamic generalization penalty reveal that frontier models find good solutions in 61 iterations but cannot escape local optima, and penalties prevent overfitting yet cap exploration when too aggressive.
\end{abstract}

\vspace{-6pt}
\section{Introduction}
\vspace{-4pt}

The Capacitated Vehicle Routing Problem (CVRP) asks for minimum-cost vehicle routes from a central depot, visiting every customer exactly once without exceeding vehicle capacity. Standard baselines include Nearest Neighbor (NN), Clarke-Wright Savings (CW) \citep{clarke1964}, and CW with 2-opt local search.

FunSearch \citep{romero2023funsearch} treats program synthesis as evolutionary search with an LLM as the mutation operator. CVRP is well-suited because construction heuristics decompose into scalar functions, greedy evaluation is fast ($O(n^2)$), and LLMs can synthesize novel feature combinations. Prior work reports training performance only; we argue this is insufficient because LLM-generated code can overfit to instance structure. We ask: \emph{Can an LLM discover CVRP heuristics that generalize?}

\textbf{Differences from original FunSearch.} We add three mechanisms absent in the original: (1) \emph{train/test/val splits} to measure generalization rather than memorization; (2) a \emph{dynamic generalization penalty} (coefficient of variation across instances) that increases when validation performance degrades; and (3) \emph{cross-dataset benchmarking} on X-set (100--1001 customers) and XL-set (1000--10,000 customers) to test transfer beyond the training distribution.

\vspace{-4pt}
\section{Method}
\vspace{-4pt}

We test two algorithmic skeletons. The \textbf{priority spec} uses greedy constructive search: a \texttt{priority(current\_node, candidate, instance, remaining\_capacity, route, route\_demand, unserved)} function scores each feasible next customer, and the highest is appended to the current route. The \textbf{savings spec} uses Clarke-Wright route merging: a \texttt{savings(i, j, instance)} function scores the benefit of merging the route ending at $i$ with the route starting at $j$. The baseline is $s_{ij} = c_{i0} + c_{0j} - c_{ij}$. Both specs provide a controlled comparison: same model, same instances, same evolution---only the skeleton differs.

Evolution uses 10 islands with behavioral clustering (per-instance score signatures) to maintain diversity. The reduced score is the score on the \emph{last} (hardest) training instance. The fitness function maximizes $\text{score} = -(\text{distance} - \text{optimal}) / \text{optimal}$; invalid solutions receive $-10^9$. Response post-processing strips markdown, collects imports, validates with \texttt{ast.parse()}, and normalizes indentation to 2-space.

Table~\ref{tab:settings} summarizes the three experiments. The Local Priority and Local Savings runs use Qwen2.5-Coder-7B on the full A-set with no split, providing a controlled spec comparison. The Cloud Priority run uses DeepSeek-v4-flash with train/val/test splits and a dynamic generalization penalty.

\section{Results}
\vspace{-4pt}

\subsection{A-Set: Spec and Model Comparison}
\vspace{-2pt}

Table~\ref{tab:overall} compares all methods on CVRPLib A-set (27 instances, 32--80 customers). The savings spec achieves 1.68\% gap---nearly $7\times$ better than the priority spec's 11.65\% gap with the \emph{same} 7B model. This confirms that the algorithmic skeleton dominates LLM capability: CW merging provides a higher-ceiling search space than greedy sequential construction.

The DeepSeek cloud run appears worse (29.8\% train gap) but three methodological differences explain this: (1) percentile\_25 reduction is stricter than last-instance; (2) the generalization penalty trades train performance for consistency; and (3) the test split removes 20\% of instances from training. Despite worse absolute numbers, the cloud run generalizes better (train 29.8\% $\to$ test 21.5\%) than the unregularized local run (train 11.7\% $\to$ test 23.7\%).

FunSearch is most effective when the initial heuristic is competent but suboptimal: too weak and search wanders; too strong and there is no headroom.

\subsection{Evolution Dynamics and Limitations}
\vspace{-2pt}

Figure~\ref{fig:evolution} shows rapid early improvement (iterations 0--400) followed by a long plateau. The Qwen priority run achieved 22 distinct milestones but stalled after iteration 650. The DeepSeek run found its best program at iteration 61---after only 62 evaluations---with no improvement thereafter. Frontier models find good solutions quickly but cannot escape local optima in program space.

\vspace{-4pt}
\section{Conclusion}
\vspace{-4pt}

We present a complete FunSearch pipeline for CVRP with train/test splits, dynamic generalization penalties, and cross-dataset benchmarking. Our experiments yield four conclusions. \textbf{First}, the algorithmic skeleton dominates LLM capability: a savings spec achieves 1.68\% gap while a priority spec plateaus at 11.65\% with the same model. \textbf{Second}, frontier models find good solutions almost immediately (best at iteration 61) but cannot escape local optima. \textbf{Third}, dynamic penalties prevent overfitting but cap at 1.0 and suppress exploration. \textbf{Fourth}, priority heuristics generalize to instances 100$\times$ larger than training, while savings heuristics overfit to small-instance structure. Code and experiments are available online.\footnote{GitHub: \url{https://github.com/laiyk5/funsearch_cvrp}; Colab: \url{https://colab.research.google.com/github/laiyk5/funsearch_cvrp/blob/main/notebooks/funsearch_cvrp.ipynb}}

\section{Spec-Specific Failure Modes}
\label{app:failure}

\textbf{Priority spec:} The greedy solver is structurally limited---it builds routes sequentially without any merge or local search phase. Even the best evolved priority function cannot escape this ceiling. On the A-set, the priority spec plateaus at 11.65\% gap while CW achieves 5.11\%. On the X-set, the gap widens: CW at 6.01\% versus evolved priority at 15.87\%. This is expected because CW is a fundamentally stronger algorithmic skeleton.

\textbf{Savings spec:} The savings spec has higher expressive power but is harder for the LLM to mutate correctly. The Qwen 7B model produced 7 milestones across 995 iterations on the A-set, achieving 1.68\% gap---nearly 7$\times$ better than the priority spec. However, the evolved savings function is brittle: small changes to the savings formula can produce wildly different merge orders, and the model frequently introduces numerical instabilities (e.g., division by zero) when trying to combine distance and demand features. The result is a function that works well on training instances but breaks down on larger problems where merge order errors compound.

\section{Dynamic Generalization Penalty Analysis}
\label{app:penalty}

The cloud run (DeepSeek-v4-flash with train/val/test splits) uses a dynamic penalty $p_t$ that starts at 0.05 and increases whenever the validation gap exceeds 5\%. The penalty trajectory is:

\begin{itemize}
\item Iterations 0--200: $p_t = 0.05$ to $0.30$. Training score improves from $-0.47$ to $-0.32$.
\item Iterations 200--480: $p_t$ rises to 1.0 and caps. Training score stalls at $-0.30$.
\item Iterations 480--550: $p_t = 1.0$ (max). Only island-specific improvements occur; no new global best.
\end{itemize}

The penalty successfully prevents train-test gap explosion (29.8\% $\to$ 21.5\% versus 11.7\% $\to$ 23.7\% without penalty). However, capping at $p_t = 1.0$ is too aggressive: the score is dominated by the penalty term, and the LLM has no incentive to propose creative mutations because any variation that improves training performance but increases cross-instance variance is immediately rejected. A staged penalty (low during exploration, high during exploitation) would likely perform better.

\section{Evolved Program Code}
\label{app:code}

\subsection{Best Priority Function (DeepSeek-v4-flash)}

The best evolved priority function (discovered at iteration 61, score $-0.2979$) incorporates a capacity-dependent demand weight and a depot-distance savings term with $\lambda = 1.5$. The key insight is that the LLM discovered---without explicit prompting---that demand weighting should be higher when the vehicle is nearly empty (first customer of a route gets weight 5.0) and lower when the vehicle is partially filled (weight scales with remaining capacity ratio).

\begin{lstlisting}
def priority(current_node: int, candidate: int, instance,
             remaining_capacity: int, route: list,
             route_demand: int, unserved: set) -> float:
  import math
  demand = instance.demands[candidate]
  if demand > remaining_capacity:
      return -float('inf')

  depot = instance.coords[0]
  cur_coords = instance.coords[current_node]
  cand_coords = instance.coords[candidate]

  d_cur = math.hypot(cur_coords[0] - cand_coords[0],
                     cur_coords[1] - cand_coords[1])
  d_dep = math.hypot(cand_coords[0] - depot[0],
                     cand_coords[1] - depot[1])

  LAMBDA = 1.5

  if current_node == 0:          # starting a new route
      distance_term = d_dep
      demand_weight = 5.0
  else:
      distance_term = d_dep - LAMBDA * d_cur
      demand_weight = 10.0 * (1.0 + remaining_capacity / instance.capacity)

  demand_term = demand / remaining_capacity if remaining_capacity > 0 else 0.0
  return distance_term + demand_weight * demand_term
\end{lstlisting}

\subsection{Best Savings Function (Qwen2.5-Coder-7B)}

The best evolved savings function (1.68\% gap on A-set) modifies the classical Clarke-Wright savings formula $s_{ij} = c_{i0} + c_{0j} - c_{ij}$ by adding demand-aware and angle-aware terms.

\begin{lstlisting}
def savings(i: int, j: int, instance) -> float:
  import math
  (x_i, y_i), (x_j, y_j), (x_d, y_d) = (
      instance.coords[i], instance.coords[j], instance.coords[0])

  d_i_depot = math.hypot(x_i - x_d, y_i - y_d)
  d_depot_j = math.hypot(x_j - x_d, y_j - y_d)
  d_i_j = math.hypot(x_i - x_j, y_i - y_j)

  if d_i_depot < 1e-9 or d_i_j < 1e-9:
      return float('inf')

  base_savings = d_i_depot + d_depot_j - d_i_j

  demand_balance_factor = (
      abs(instance.demands[i] + instance.demands[j]) / instance.capacity)

  avg_distance = (d_i_depot + d_depot_j) / 2.0
  spatial_proximity_factor = min(1.0, max(d_i_depot, d_depot_j) / avg_distance)

  angle_i_depot_j = (math.atan2(y_j - y_d, x_j - x_d)
                     - math.atan2(y_i - y_d, x_i - x_d))
  if angle_i_depot_j < 0:
      angle_i_depot_j += 2 * math.pi

  collinearity_factor = min(1.0, abs(math.sin(angle_i_depot_j)) / (math.pi / 4))

  direction_i_depot = math.atan2(y_i - y_d, x_i - x_d)
  direction_depot_j = math.atan2(y_j - y_d, x_j - x_d)
  direction_diff = abs(direction_i_depot - direction_depot_j)
  if direction_diff > math.pi:
      direction_diff = 2 * math.pi - direction_diff
  alignment_factor = min(1.0, abs(math.cos(direction_diff)))

  route_length_penalty = max(d_i_depot, d_depot_j) / instance.capacity
  if route_length_penalty > 2.0:
      route_length_penalty = 1.0 + (route_length_penalty - 2.0) * 0.5

  curvature_penalty = abs(math.tan(angle_i_depot_j)) / instance.capacity
  if curvature_penalty > 2.0:
      curvature_penalty = 1.0 + (curvature_penalty - 2.0) * 0.5

  segment_count_penalty = max(1.0, d_i_depot + d_depot_j) / instance.capacity
  if segment_count_penalty > 4.0:
      segment_count_penalty = 1.0 + (segment_count_penalty - 4.0) * 0.5

  long_segment_penalty = max(
      0, max(d_i_depot, d_depot_j) - instance.capacity / 2) / instance.capacity

  return (base_savings
          + demand_balance_factor * 3
          + spatial_proximity_factor * 4
          + collinearity_factor * 5
          + alignment_factor * 3
          - route_length_penalty * 1.5
          - curvature_penalty * 2
          - segment_count_penalty * 1
          - long_segment_penalty * 2)
\end{lstlisting}
