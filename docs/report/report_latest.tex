\begin{abstract}
We present a FunSearch implementation for automatically discovering CVRP construction heuristics. Using code-generation models from 7B to frontier-scale, we evolve priority and savings functions across greedy and Clarke-Wright solvers. A controlled comparison reveals that the algorithmic skeleton matters more than LLM capability: a savings spec achieves 1.68\% gap while a priority spec plateaus at 11.65\% gap with the same model. With train/test/val splits and a dynamic generalization penalty, a frontier model finds its best solution in 61 iterations but the penalty caps at 1.0 and suppresses further exploration. Our results show that (1) strong baselines provide higher-ceiling improvement, (2) generalization penalties can prevent overfitting but may be too aggressive, and (3) LLM-driven search discovers useful heuristics quickly but struggles to escape local optima in program space.
\end{abstract}

\vspace{-6pt}
\section{Motivation and Problem}
\vspace{-4pt}

The Capacitated Vehicle Routing Problem (CVRP) asks for minimum-cost vehicle routes from a central depot, visiting every customer exactly once without exceeding vehicle capacity. As an NP-hard problem, exact methods scale poorly and practitioners rely on heuristics such as Nearest Neighbor, Clarke-Wright Savings \citep{clarke1964}, and Sweep \citep{gillet1974}---all hand-designed by domain experts.

\textbf{Why automate heuristic discovery?} Designing effective heuristics requires deep domain expertise and trial-and-error. FunSearch \citep{romero2023funsearch} offers a compelling alternative: treat program synthesis as evolutionary search, using an LLM as a mutation operator. This is particularly appropriate for CVRP because (1) construction heuristics naturally decompose into scalar priority functions, providing a compact search space; (2) greedy evaluation is fast ($O(n^2)$ per instance), enabling thousands of iterations; and (3) LLMs can synthesize novel feature combinations that human designers might overlook.

\textbf{Why is this hard?} Prior work on LLM-driven algorithm discovery \citep{romero2023funsearch} reports training performance only. We argue this is insufficient: LLM-generated code can easily overfit to training instance structure, and without held-out evaluation, one cannot distinguish learning routing \emph{principles} from memorizing \emph{structure}. Our core question is: \emph{Can an LLM discover CVRP heuristics that generalize?}

\section{Method}
\vspace{-4pt}

\subsection{Why a Greedy Priority Function?}
\vspace{-2pt}

A CVRP instance is $I = (V, c, d, Q)$ where $V = \{0, 1, \dots, n\}$ includes the depot (0), $c_{ij}$ is Euclidean distance, $d_i$ is demand, and $Q$ is capacity. We define a greedy solver that repeatedly calls a \texttt{priority()} function to choose the next customer:
\begin{verbatim}
def priority(current_node, candidate, instance,
             remaining_capacity, route, route_demand,
             unserved) -> float
\end{verbatim}
Higher values indicate stronger preference. This design is deliberate: the search space is rich enough to encode sophisticated strategies (distance, capacity, depot proximity, demand density) yet compact enough that a 7B model can propose meaningful mutations in a few tokens.

The fitness function maximizes $\text{score} = -(\text{distance} - \text{optimal}) / \text{optimal}$, so $-0.10$ corresponds to a 10\% gap above optimal. Invalid solutions receive $-10^9$.

\subsection{Why Island-Based Evolution?}
\vspace{-2pt}

We use 10 islands, each maintaining a population clustered by behavioral signature (per-instance score vectors). This is appropriate because different CVRP instances reward different routing strategies: dense urban instances favor short-distance routing, while sparse rural instances favor depot-proximity. Island diversity prevents premature convergence to a single strategy that works well on only a subset of cases. Following the DeepMind reference, the reduced score for a program is the score on the \emph{last} training instance (the hardest), not the mean or maximum. This biases search toward robust performance on difficult cases.

\subsection{Train/Test Split for Generalization}
\vspace{-2pt}

We hold out 5 of 27 CVRPLib A-set instances as a test set. Measuring generalization is critical because LLM-generated heuristics might exploit idiosyncrasies of training instances. Without held-out evaluation, reported improvements may be illusory. We evaluate the best program on the test set every 100 iterations.

\subsection{What the Evolved Function Reveals}
\vspace{-2pt}

The best evolved \texttt{priority()} function incorporates: Euclidean distance to candidate, distance to depot weighted by remaining capacity, demand penalty, route-length penalty, proximity to unserved customers, and customer density near the current node. Crucially, the LLM synthesized these features \emph{without explicit prompting}, demonstrating emergent feature engineering. The function blends distance-minimization with capacity-balancing---a non-obvious combination.

\subsection{The Overfitting Insight}
\vspace{-2pt}

Table~\ref{tab:generalization} compares generalization across the local run (no regularization) and the cloud run (with dynamic penalty). The local run shows severe overfitting: training gap doubles on test. The cloud run with generalization penalty achieves a smaller train-to-test gap (29.8\% $\to$ 21.5\%), but at the cost of much worse absolute performance---the penalty suppresses exploration and the search cannot find solutions competitive with the unregularized local run.

This has methodological implications. Prior work on LLM-driven algorithm discovery often reports training performance only; our train/test split demonstrates that this can be misleading. However, na\"{i}ve regularization (a uniform CV penalty) may be too blunt: it prevents overfitting but also prevents discovery of good training solutions. A more sophisticated approach would use cross-validation during scoring or only apply the penalty after an initial exploration phase.

\subsection{Spec Comparison: Priority vs.\ Savings}
\vspace{-2pt}

A controlled comparison between the two algorithmic specifications (same Qwen2.5-Coder-7B model, same instances, no train/test split) reveals that the \emph{search space} matters more than the LLM's mutation quality. Table~\ref{tab:speccompare} shows that the savings spec achieves a 1.68\% gap---nearly 7$\times$ better than the priority spec's 11.65\% gap. This confirms a key design principle in learned heuristic search: a strong algorithmic skeleton (Clarke-Wright merging) provides higher-ceiling improvement than a weak skeleton (greedy sequential construction), even when both are evolved by the same LLM.

The savings spec produced only 7 milestones across 995 iterations---the starting Clarke-Wright heuristic is already strong, leaving limited headroom for LLM-driven improvement. In contrast, the priority spec had 0 milestones in this run because the initial nearest-neighbor heuristic is so weak that almost any mutation is accepted, but acceptance does not correlate with improvement. This suggests that FunSearch is most effective when the initial heuristic is competent but suboptimal: too weak and search wanders aimlessly; too strong and there is little room to improve.

\subsection{Cloud API Run: DeepSeek-v4-flash with Generalization Penalty}
\vspace{-2pt}

Switching to the more capable DeepSeek-v4-flash model and adding train/test/val splits (15\% validation, 20\% test) with a dynamic generalization penalty yields qualitatively different behavior. After 550 iterations, the best training score is $-0.2979$ (29.8\% gap), which appears worse than the local-model runs. However, this is explained by three methodological differences: (1) the score reduction method changed from ``last'' to ``percentile\_25'', making the metric stricter; (2) a generalization penalty (coefficient of variation across instances) is subtracted from the score, directly trading train performance for consistency; and (3) the test split removes instances from training, reducing the training set size.

\textbf{Validation feedback dynamics.} The dynamic penalty starts at 0.05 and increases whenever the validation gap exceeds 5\%. It capped at 1.0 by iteration 480 and has remained there since. The validation gap stabilized at 0.070 (train clean $-0.2743$ vs.\ val $-0.3444$), but the test score is $-0.2147$ (21.5\% gap). This reveals two insights: first, the penalty successfully prevents the train-test gap from exploding (old run: 11.7\% $\to$ 23.7\%; new run: 29.8\% $\to$ 21.5\%); second, the penalty may be \emph{too aggressive}---capping at 1.0 suppresses exploration, and the run has found only 1 truly new best since iteration 61 (out of 62 milestones, only 1 reached $-0.2979$; the rest are island-specific improvements at much worse scores).

\textbf{Sample efficiency.} The best program ($-0.2979$) was discovered at iteration 61---after only 62 evaluations. The subsequent 490 evaluations produced no improvement. This is both encouraging (the capable model finds good solutions quickly) and concerning (the search enters a deep local optimum and cannot escape). The evolved program is 56 lines---more compact than the Qwen-evolved programs (87--83 lines)---suggesting DeepSeek generates cleaner, more focused mutations.

\textbf{Limitations and future work.} The gap to Clarke-Wright Savings remains large (11.7\% vs.\ 5.1\%). We attribute this to two factors: (1) the constructive solver is strictly less expressive than route-merging algorithms, and (2) the 7B model may lack capacity for more sophisticated algorithmic structures. Directions include evolving improvement operators (e.g., 2-opt, relocate), stronger regularization such as cross-validation during scoring, and evolving multi-stage algorithms rather than single priority functions.

\vspace{-4pt}
\section{Conclusion}
\vspace{-4pt}

We present a complete FunSearch pipeline for CVRP with engineering contributions in observability, resumability, and generalization measurement. Our experiments yield four conclusions. \textbf{First}, the algorithmic skeleton dominates LLM capability: a savings spec achieves 1.68\% gap while a priority spec plateaus at 11.65\% with the same model. \textbf{Second}, frontier models find good solutions almost immediately (best at iteration 61) but cannot escape local optima thereafter. \textbf{Third}, dynamic generalization penalties prevent train-test gap explosion but cap at 1.0 and suppress exploration. \textbf{Fourth}, LLM-driven heuristic search is most effective when the initial heuristic is competent but suboptimal---too weak and search wanders; too strong and there is no headroom. Code and experiments are available at \url{https://github.com/laiyk5/funsearch_cvrp}.
